# 自动化规则

自动化规则允许将设备事件、数据变化或定时触发器与一系列动作关联，实现自动化的设备控制和工作流程，无需人工干预。

## 核心概念

### 触发器类型

| 类型 | 说明 | 示例 |
|------|------|------|
| 事件触发 (event) | 设备数据变化或告警触发 | 温度超过 30°C |
| 定时触发 (schedule) | 按 Cron 表达式定时执行 | 每天 8:00 执行 |
| 手动触发 (manual) | 由用户手动触发 | 点击按钮执行 |

### 动作类型

| 类型 | 说明 | 配置字段 |
|------|------|----------|
| device_command | 向设备发送命令 | device_id, command |
| notification | 发送通知 | channel_id, message |
| http_request | 发送 HTTP 请求 | url, method, body |
| automation | 触发另一个自动化 | automation_id |

## 创建自动化规则

### 示例一：温度超过阈值自动关空调

1. 进入「自动化」→「规则列表」
2. 点击「添加规则」
3. 填写基本信息：
   - **名称**：温度过高关闭空调
   - **描述**：当温度超过 30°C 时自动关闭空调
   - **触发类型**：事件触发

4. 配置触发条件：
   - **事件源类型**：设备
   - **设备**：温度传感器 01
   - **属性**：temperature
   - **条件**：>
   - **阈值**：30
   - **持续时间**：60 秒（避免抖动）

5. 配置执行动作：
   ```json
   [
     {
       "type": "device_command",
       "device_id": "device_ac_001",
       "command": "off"
     }
   ]
   ```

6. 设置高级选项：
   - **超时时间**：30 秒
   - **重试次数**：3 次
   - **冷却时间**：300 秒（避免频繁触发）
   - **优先级**：100

7. 保存规则

### 示例二：定时开关灯

1. 新建规则：
   - **名称**：定时开关灯
   - **触发类型**：定时触发
   - **Cron 表达式**：`0 8,22 * * *`（每天 8:00 和 22:00）

2. 执行动作：
   ```json
   [
     {
       "type": "device_command",
       "device_id": "device_light_001",
       "command": "toggle"
     }
   ]
   ```

### 示例三：设备离线自动通知

1. 新建规则：
   - **名称**：设备离线通知
   - **触发类型**：事件触发
   - **事件类型**：offline

2. 配置触发条件：
   - **事件源类型**：设备
   - **设备**：选择要监控的设备

3. 执行动作：
   ```json
   [
     {
       "type": "notification",
       "channel_id": "channel_email_001",
       "message": "设备 {device_name} 已离线"
     }
   ]
   ```

## 管理自动化规则

### 启用/禁用规则

点击规则列表中的开关按钮，可随时启用或禁用规则。

### 测试规则

在正式启用前，可使用测试功能验证规则配置：

```http
POST /api/v1/automations/{id}/test
Content-Type: application/json

{
  "conditions": "{\"property\":\"temperature\",\"operator\":\">\",\"value\":30}",
  "mock_data": {
    "temperature": 35
  }
}
```

### 手动执行规则

```http
POST /api/v1/automations/{id}/run
```

### 查看执行统计

每个规则显示：
- **总执行次数**：规则被触发的总次数
- **成功次数**：成功执行所有动作的次数
- **失败次数**：执行失败的次数
- **成功率**：成功次数 / 总执行次数
- **最后执行时间**：最近一次执行的时间
- **最后执行状态**：success / failed / running

## Cron 表达式

### 基本格式

```
┌───────────── 秒 (0-59)
│ ┌───────────── 分钟 (0-59)
│ │ ┌───────────── 小时 (0-23)
│ │ │ ┌───────────── 日 (1-31)
│ │ │ │ ┌───────────── 月 (1-12)
│ │ │ │ │ ┌───────────── 星期 (0-6，0=周日)
│ │ │ │ │ │
* * * * * *
```

### 常用表达式

| 表达式 | 说明 |
|--------|------|
| `0 * * * *` | 每小时整点 |
| `0 0 * * *` | 每天午夜 |
| `0 8 * * *` | 每天早上 8:00 |
| `*/5 * * * *` | 每 5 分钟 |
| `0 8-18 * * *` | 每天 8 点到 18 点每小时 |
| `0 9 * * 1-5` | 工作日 9:00 |
| `30 8 1 * *` | 每月1号 8:30 |

## 冷却时间

冷却时间（Cooldown）用于防止规则在短时间内被重复触发：

| 场景 | 推荐冷却时间 |
|------|-------------|
| 温度控制（空调） | 5-10 分钟 |
| 灯光控制 | 1-2 分钟 |
| 通知类 | 0（可重复通知） |
| 设备命令 | 30-60 秒 |

## 条件组合

支持在触发条件中使用复杂的 AND/OR 逻辑：

```json
{
  "conditions": [
    { "property": "temperature", "operator": ">", "value": 30 },
    { "property": "humidity", "operator": "<", "value": 40 }
  ],
  "logic": "OR"
}
```

## 常见问题

**Q：规则触发了但动作没有执行？**
- 检查动作配置是否正确
- 查看规则执行日志排查具体失败原因
- 确认目标设备在线且驱动支持该命令

**Q：如何避免规则被频繁触发？**
- 设置合理的冷却时间（cooldown）
- 使用持续时间（duration）过滤短暂的状态变化

**Q：多个规则同时触发同一设备会冲突吗？**
- 按优先级（priority）顺序执行，数字越小优先级越高
- 建议合理规划规则，避免冲突
