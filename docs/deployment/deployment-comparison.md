# 部署方案对比

## 方案对比

### 方案 A: 分离部署（当前）

```
┌─────────────┐      ┌─────────────┐
│  Next.js    │      │   Axum      │
│  (Node.js)  │─────▶│  (Rust)     │
│  Port 3001  │      │  Port 3002  │
└─────────────┘      └─────────────┘
```

**优点**:
- 开发体验好（热重载）
- 前后端独立开发
- 可以使用 Next.js SSR 特性

**缺点**:
- 需要 Node.js 运行时
- 需要管理两个进程
- 部署复杂（需要 nginx 反向代理）
- 资源占用高（Node.js + Rust）

### 方案 B: 静态单二进制（推荐）

```
┌──────────────────────────────┐
│     单个二进制文件            │
│  ┌────────────────────────┐  │
│  │  嵌入的静态文件         │  │
│  └────────────────────────┘  │
│  ┌────────────────────────┐  │
│  │  Axum (API + 静态)     │  │
│  └────────────────────────┘  │
└──────────────────────────────┘
```

**优点**:
- 单文件部署
- 无需 Node.js
- 资源占用低
- 启动快速
- 适合嵌入式设备

**缺点**:
- 无 SSR（但本项目不需要）
- 构建时间稍长
- 调试稍复杂

## 资源对比

| 指标 | 分离部署 | 静态单二进制 |
|------|---------|-------------|
| 内存占用 | ~200MB | ~50MB |
| 磁盘占用 | ~500MB | ~40MB |
| 启动时间 | ~5s | <1s |
| 部署文件数 | 1000+ | 1 |
| 运行时依赖 | Node.js + Rust | 仅 Rust |

## 适用场景

### 使用静态单二进制
- ✅ 嵌入式设备（ARM）
- ✅ 容器化部署
- ✅ 边缘计算
- ✅ 资源受限环境
- ✅ 简化运维

### 使用分离部署
- ✅ 需要 SSR
- ✅ 需要 ISR
- ✅ 复杂的前端路由
- ✅ 开发阶段

## 迁移建议

### 当前项目评估

检查项目是否使用了 Next.js 服务端特性：

```bash
# 搜索服务端 API
grep -r "getServerSideProps\|getStaticProps\|cookies()\|headers()" web/app/
```

**结果**: 未使用 ✅

1. **测试静态导出**

   - 检查所有页面是否正常
   - 验证 API 调用
   - 测试路由跳转

3. **构建单二进制**
   ```bash
   .\scripts\build-static.ps1 -Release
   ```

4. **部署测试**
   ```bash
   # 复制到测试环境
   scp api/target/release/tinyiothub test-server:/opt/
   
   # 启动测试
   ssh test-server
   /opt/tinyiothub
   ```

## 推荐方案

**对于 TinyIoTHub 项目，强烈推荐使用静态单二进制方案**

理由：
1. 目标设备是嵌入式 ARM 设备
2. 不需要 SSR 特性
3. 简化部署和运维
4. 降低资源占用
5. 提高启动速度

## 回滚方案

如果静态单二进制方案出现问题，可以快速回滚：

```bash
# 1. 停止单二进制服务
systemctl stop tinyiothub

# 2. 启动分离服务
cd /opt/tinyiothub
pm2 start web/ecosystem.config.js
systemctl start tinyiothub-api

# 3. 恢复 nginx 配置
cp nginx.conf.backup /etc/nginx/sites-enabled/tinyiothub
nginx -s reload
```
