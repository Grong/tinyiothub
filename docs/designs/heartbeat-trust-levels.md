# CEO Plan: Heartbeat Trust Levels — AI Autonomous Execution

## Vision

### 10x Check

A truly autonomous IoT operations center. AI monitors devices, diagnoses anomalies, and
executes repairs — humans only see a morning digest of what happened overnight, and only
get interrupted when AI explicitly says "I need your judgment on this one."

Current state: AI generates a pile of approval requests for every routine operation,
turning the "AI assistant" into a "human approval bottleneck." This is worse than not
having AI — it creates work instead of removing it.

The trust-level system inverts this: the human tells AI what it's allowed to do, AI
executes autonomously within that boundary, and the human audits via history and
rollback. Trust becomes a conscious configuration choice, not an AI guess.

### Platonic Ideal (12-month target)

```
Morning. User opens dashboard. One notification:

  "昨夜 AI 巡检摘要:
   - 自动重连 3 台离线设备 ✓
   - 确认并处理 12 条告警 ✓
   - 调整 2 台设备的温度阈值（因环境变化）✓
   - 发现 dev-042 反复重连失败，疑似硬件故障 —— 需要你确认是否派工单
   - 所有操作详情 → 执行历史"

Zero approval requests. One decision needed. Trust earned through transparency.
```

NOTE: The morning digest UX described above is aspirational. The MVP delivers trust
configuration, enforcement, and rollback mechanisms. The digest UX is a future iteration
built on the existing AI patrol summary reports.

## Architecture Decisions

### Trust enforcement model: prompt-guided OR hard-enforced?

The current system uses prompt-injected L0-L3 severity guidance at
`cloud/src/modules/agent/heartbeat.rs:421-432`. The LLM reads these guidelines and
self-regulates — choosing whether to auto-execute or propose.

The new system replaces this with hard enforcement. The prompt no longer contains
L0-L3 classification. Instead, the prompt lists tools available at each trust level,
and the LLM is instructed to only use tools at its configured level. The code-level
enforcement happens in the tool adapter layer:

```
LLM calls tool → tool adapter checks trust config → allow / deny / log-and-allow
```

If a tool is not at the configured trust level, it is blocked at the adapter (not at
the prompt level), preventing the LLM from accidentally or intentionally exceeding
its trust boundary.

The existing `tool_denylist: Vec<String>` in `AgentRuntimeConfig` (at
`cloud/src/shared/agent/config.rs:80`) is **subsumed** by the new trust config:
`disabled` trust level = denylist entry. Migration: existing denylist entries become
`disabled` trust levels for those tools.

### Enforcement interception point

The heartbeat loop uses zeroclaw's **streaming `run()` API** (same as chat flow at
`chat/service.rs:25-59`), which emits `TurnEvent::ToolCall` events in real time.
Before dispatching each tool call, the loop checks trust_config for the
(tool_name, device_id) pair and decides: allow, log-and-allow, or deny.

**Data flow:**
```
heartbeat_loop() tick start:
  │
  ├─ 1. Load trust_config from workspaces.heartbeat_trust_config (per-tick DB read)
  │     │
  │     └─ Snapshot config for this tick (mid-tick changes don't affect current run)
  │
  ├─ 2. Build prompt with trust-level-tagged tool list
  │
  ├─ 3. agent_pool.run() streaming — TurnEvent::ToolCall emitted by zeroclaw
  │     │
  │     └─ Before dispatch: check trust_config[tool][device_id] or [tool]["*"]
  │           │
  │           ├─ full_auto → dispatch + record action_type="auto_executed"
  │           ├─ auto_with_log → dispatch + record action_type="auto_executed" (logged)
  │           ├─ approval_required → BLOCK, return error to LLM:
  │           │     "Tool X is blocked. Propose it in your HealingReport as pending_proposal."
  │           └─ disabled → BLOCK silently, record action_type="trust_blocked"
  │
  └─ 4. Parse HealingReport from LLM text response
        │
        └─ Cross-reference: TurnEvent tool calls vs HealingReport auto_executed
              ├─ Match → normal
              ├─ TurnEvent but not in HealingReport → flag "unreported_tool_call"
              └─ HealingReport but not in TurnEvent → flag "hallucinated_tool_call"
```

**Chat vs heartbeat distinction:** Trust enforcement is skipped when the agent is
NOT the `__heartbeat__` agent (agent_id check). Chat agents continue to use
zeroclaw's built-in `AutonomyLevel::Supervised` with no trust_config gating.

**Config snapshot per tick:** trust_config is loaded from DB at tick start and
held for the duration of the tick. Mid-tick config changes via REST API take
effect on the next tick.

**Trust enforcement injected via `load_all_tools()`:** An `Arc<DashMap<String,
TrustConfigMap>>` parameter is added to `load_all_tools()` and threaded through
to each `IoTToolAdapter` instance. The `execute()` method reads the shared map
using `self.workspace_id` for the workspace key. Device_id is extracted from
tool call args (all IoT tools include a `device_id` field; tools without it
use the `"*"` wildcard only).

**Error handling:** If trust_config DB load fails at tick start, the tick is
skipped and the error is logged. No tools are called without trust config.

### Persistence

`HeartbeatConfig` is managed per-workspace via `HeartbeatManager.configs:
DashMap<String, HeartbeatConfig>` (heartbeat_manager.rs:64). The per-tool trust
config is stored as a new JSON column `heartbeat_trust_config TEXT` added to the
existing `workspaces` table via migration.

**Migration SQL:**
```sql
ALTER TABLE workspaces ADD COLUMN heartbeat_trust_config TEXT NOT NULL DEFAULT '';

-- Backfill: migrate existing tool_denylist entries to disabled trust level
-- Reads agent_config JSON, extracts tool_denylist, writes to heartbeat_trust_config
-- Handled in Rust migration code (not raw SQL — JSON manipulation needed)
```

**Backfill logic (Rust migration):** For each workspace with a non-empty `agent_config`
containing `tool_denylist: ["tool_a", "tool_b"]`, set
`heartbeat_trust_config = '{"tool_a":{"*":"disabled"},"tool_b":{"*":"disabled"}}'`.

**Why the `workspaces` table:** The `workspace_configs` table does not exist in the
current schema. The closest existing column is `agent_config TEXT` in `workspaces`
(migration 20260405074640). Adding `heartbeat_trust_config` to `workspaces` follows
this pattern — workspace-scoped config stored as a JSON text column.

**Per-workspace isolation:** `HeartbeatManager` already uses a `DashMap<String,
HeartbeatConfig>` keyed by workspace_id (heartbeat_manager.rs:64). The trust_config
is loaded from `workspaces.heartbeat_trust_config` at startup into the per-workspace
HeartbeatConfig entry. The REST API `PUT /workspaces/{id}/heartbeat/trust-config`
both persists to DB and updates the in-memory DashMap entry.

Default for existing workspaces after migration: `heartbeat_trust_config = ''` →
empty config → all tools default to `approval_required` (safe default — no change
in behavior until user explicitly configures).

### Before-snapshot capture for rollback

The LLM cannot reliably report before-values in its text response. Instead, the
tool adapter layer captures snapshots:

1. When a tool call (e.g. `write_properties`) is about to execute, the adapter reads
   the current property values via `read_properties` and stores them as
   `before_snapshot` in the agent_action record.
2. For tools that are NOT state-mutating or NOT reversible (e.g. `send_command("reboot")`,
   `alarm_acknowledge`), rollback is disabled. The UI shows "不可回滚" for these actions.

Rollback capability per tool:
| Tool | Rollback? | Mechanism |
|------|-----------|-----------|
| write_properties | Yes | Reverse-write before_snapshot values |
| send_command | No | Commands are non-reversible side effects |
| alarm_acknowledge | Partial | Un-acknowledge the alarm (if API supports it) |
| search_devices / get_device / read_properties | N/A | Read-only, no rollback needed |

### Auto-learning pattern definition

A "pattern" is defined as `(tool_name, device_id)` pair. When a proposal for the
same (tool, device) pair is approved >= 3 times, the system surfaces a suggestion:
"AI 已多次对设备 X 执行 tool_name 操作并被批准，要设为自动吗？"

This is deliberately simple. Semantic pattern matching on summary text is unreliable.
The device-level granularity matches how users think about trust: "I trust AI to
manage THIS specific device, not all devices."

## Scope Decisions

| # | Proposal | Effort | Decision | Reasoning |
|---|----------|--------|----------|-----------|
| 1 | 工具级信任粒度 | M | ACCEPTED | 比全局开关实用 10x——用户可能信任 AI 发重连命令但不信任它改属性值 |
| 2 | 一键回滚 | M | ACCEPTED | 回滚是信任的前提。没有后悔药，用户不敢放权 |
| 3 | 信任校准向导 | S | ACCEPTED | 消除"盲目选择"焦虑，30 分钟投入换用户信心 |
| 4 | 自动审批学习 | S | ACCEPTED | 简化为 (tool, device_id) 频次统计，长期减少重复审批 |

## Accepted Scope

### 1. 工具级信任粒度（支持 per-device 覆盖）
- **Backend**: 在 `HeartbeatConfig`（heartbeat_manager.rs）中新增字段
  `trust_config: HashMap<String, HashMap<String, TrustLevel>>`（tool_name → device_id → level）
  其中 `device_id = "*"` 作为通配符匹配所有设备，device_id = 具体设备 ID 覆盖通配符
- **TrustLevel enum**: `full_auto` | `auto_with_log` | `approval_required` | `disabled`
- **优先级规则**: device-specific 配置 > `"*"` 通配符 > 默认 `approval_required`
- **Enforcement**: 工具适配器层（tools/service.rs）在 dispatch 前检查 trust_config，先查 tool:device 精确匹配，再查 tool:* 通配符
- **Migration**: 已有 `tool_denylist` 条目自动映射为 `tool_name → "*" → disabled`
- **API**: `GET/PUT /workspaces/{id}/heartbeat/trust-config` — 读写信任配置
- **Frontend**: 在心跳配置卡片中新增"工具信任"section，每项工具一个下拉选择器，支持展开 per-device 覆盖
- **Fallback**: 未配置的工具默认 `approval_required`（安全优先）

### 2. 一键回滚
- **Snapshot 捕获**: 工具适配器在 write 类操作执行前自动读取当前值存入 `before_snapshot`
- **agent_actions**: content JSON 新增 `before_snapshot` 字段（仅 auto_executed 类型）
- **回滚机制**: `write_properties` → 反向写入 before_snapshot 值
  `alarm_acknowledge` → 调用 unacknowledge API（如果支持）
  `send_command` → 不支持回滚，UI 标注"不可回滚"
- **前端**: 执行历史中每条 auto_executed 记录，可回滚的显示"回滚"按钮
- **回滚记录**: 回滚操作也记录为 agent_action（action_type="rollback"），关联原 action

### 3. 信任校准向导
- **触发条件**: 工作空间首次保存信任配置时（trust_config 为空或全部为默认值）
- **展示内容**: 基于最近 24h 的 proposal 记录，模拟在不同信任级别下的行为差异
  - 列出哪些操作会在"激进"模式下自动执行
  - 列出哪些操作仍需审批
- **实现**: 前端纯计算——获取历史 proposal 列表，按 tool_name 分组，对照信任级别映射
- **非侵入**: 不改变任何后端逻辑，仅前端 UI

### 4. 自动审批学习
- **模式定义**: `(tool_name, device_id)` 二元组
- **阈值**: 同一 (tool, device) 的 proposal 被 approve ≥ 3 次
- **行为**: 触发后，在下次心跳结果摘要中追加提示："检测到你对 dev-X 的 tool_name 操作已批准 3 次，要设为自动执行吗？[一键设置]"
- **确认后**: 更新 trust_config，将该工具的该设备设为 `full_auto`
- **存储**: proposal 的 approve/reject 状态已在 agent_actions content JSON 中，无需新表

## NOT in Scope
- 不引入新的 AI 模型做风险评估
- 不实现分布式信任策略（多用户/多角色）
- Daily digest 邮件/通知（已有 AI 巡检日报的基础，后续迭代）
- Morning digest UX 是未来迭代目标，不在本次 MVP

### Test Strategy
- Unit tests: trust_config lookup logic (priority: device > "*" > default, wildcard matching, empty config)
- Unit tests: TrustLevel enum serialization/deserialization
- Integration test: heartbeat tick with mock trust_config — verify blocked tools are denied, allowed tools execute, HealingReport cross-reference works
- No E2E tests for MVP (frontend UI verification done manually)

### Observability
- Trust enforcement decisions logged: `tracing::info!(tool = %tool, device = %device_id, level = ?level, decision = %decision, "trust check")`
- Blocked tool calls recorded as agent_action with action_type="trust_blocked"
- Rollback operations recorded with action_type="rollback" and reference to original action

### Rollback (deployment rollback, not feature rollback)
- New `trust_config` field defaults to empty → all tools `approval_required`
- Existing heartbeat loops continue unchanged until user explicitly configures trust
- New API endpoints are additive, no breaking changes to existing routes
